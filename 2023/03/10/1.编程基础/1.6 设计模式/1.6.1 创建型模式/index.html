    1.6.1【编程基础 - 设计模式】创建型模式

    创建型模式：对类的实例化过程进行了抽象，能够将软件模块中 对象的创建 和对象的使用分离

    一、简单工厂模式

    1.1 意图

    • 调用者与创建者解藕，调用者不需要知道创建细节

    1.2 主要解决

    • 调用者实例化对象时的耦合问题

    1.3 何时使用

    • 调用者在不同条件会创建不同的实例
    • 工厂中的产品较为固定时使用

    1.4 注意

    • 如果使用简单工厂模式，就需要引进一个简单工厂类，会增加项目复杂度
    • 并且此工厂的所有车间都已经被划分给固定产品，在添加产品时需要扩大车间（修改工厂类）

    1.5 示例代码

    代码非常简单，但对添加产品非常不友好，需要修改代码

    1.6 开源例子

    • Spring 框架中的 BeanFactory 就是一个简单工厂，它负责创建和管理应用程序中的 Bean。当应用程序需要一个 Bean 时，只需要调用 BeanFactory 的 getBean() 方法，并传入 Bean 的名称即可
    • Log4j 是一个流行的 Java 日志框架。在 Log4j 中，Logger 对象的创建就是使用了简单工厂模式。LoggerFactory 是一个简单工厂，它根据传入的类名创建相应的 Logger 对象，以便记录日志

    二、工厂方法模式

    2.1 意图

    • 调用者与创建者解藕，调用者不需要知道创建细节

    2.2 主要解决

    • 调用者实例化对象时的耦合问题

    2.3 何时使用

    • 调用者在不同条件会创建不同的实例时
    • 工厂中的产品在不断变化时

    2.4 注意

    • 一个工厂只能创建一个具体的产品
    • 添加新的产品时，除了需要添加新的产品类外，还需要提供与之对应的具体工厂类，类的个数会成对增加，增加了项目的复杂度

    三、抽象工厂模式

    3.1 意图

    • 调用者与创建者解藕，调用者不需要知道创建细节

    3.2 主要解决

    • 调用者实例化对象时的耦合问题

    3.3 何时使用

    • 需要同时创建一批比较固定的、彼此之间有关联产品时

    3.4 注意

    • 抽象工厂很难支持添加新种类产品，因为抽象工厂中已经确定了被创建的产品族中包含的产品

    3.5 简单工厂和工厂方法的问题

    • 对于需要同时创建一批比较固定的、彼此之间有关联产品时，简单工厂和工厂方法就不适用了，它们没有解决对象之间互相依赖匹配的问题

    3.6 解决方案

    • 将需要同时创建的产品抽象出一个产品族

    四、单例模式

    4.1 意图

    • 保证一个类仅有一个实例，并提供一个访问它的全局访问点

    4.2 何时使用

    • 需要控制实例的数量，节省系统资源的时候

    4.3 注意

    • 单例类只能有一个实例
    • 单例类必须自己创建自己的唯一实例
    • 单例类必须给所有其他对象提供这一实例

    4.4 饿汉式 - 线程不安全

    在以下实现中，私有静态变量 instance 被延迟实例化。这样做的好处是，如果没有用到该类，那么就不会实例化 instance，从而节省资源

    4.5 饿汉式

    直接实例化 instance 保证了线程安全，但是直接实例化的方式丢失了延迟实例化带来的节约资源的好处

    4.6 双重检测锁

    先判断 instance 是否已经被实例化，如果没有实例化，则进入加锁语句块再次判断 instance 是否已经被实例化，若没有被实例化则进行实例化，注意 instance 需要被 volatile 修饰保证获取到的是最新数据

    4.7 静态内部类 - 线程安全（推荐使用）

    当 SingleCase 被加载时，静态内部类 StaticInsideClass 并没有被加载进内存。只有调用 getInstance() 方法时才会实例化 StaticInsideClass 内的 INSTANCE，并且 JVM 能确保 INSTANCE 只被实例化一次

    4.8 枚举 - 线程安全

    简洁，自动支持序列化机制，绝对防止多次实例化

    4.9 开源例子

    • Spring 中 ApplicationContext 和 BeanFactory 都是单例对象。在 Spring 中，容器（如 ApplicationContext 或 BeanFactory）负责创建和管理 Bean 对象。为了确保容器对 Bean 的管理是有效和高效的，Spring 默认会将容器本身定义为单例模式
    • Log4j 是一个流行的日志框架，它使用单例模式来管理 Logger 对象。在 Log4j 中，Logger 对象是单例的，它可以被多个线程共享，以实现高效的日志记录

    五、建造者模式

    5.1 意图

    简化类的构造方式，避免一堆参数的构造方法，避免使用一堆 getter / setter 方法

    5.2 何时使用

    当一个类的构造函数参数个数超过 4 个，而且这些参数有些是可选的参数，考虑使用构造者模式

    七、总结

    • 简单工厂模式：通过一个工厂类来创建产品对象，将对象的创建和使用分离，客户端只需知道需要什么产品即可
    • 工厂方法模式：将工厂抽象出来，定义一个工厂接口，由具体的工厂类来实现工厂接口，客户端通过调用工厂接口来创建产品对象
    • 抽象工厂模式：提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们的具体类。使用抽象工厂可以帮助我们在不必指定具体类的情况下创建一系列相关或相互依赖的对象
    • 单例模式：保证一个类只有一个实例，并提供一个全局访问点，使得这个实例可以被客户端访问
    • 建造者模式：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。通常由一个指挥者与若干个建造者组成，指挥者负责协调建造者完成构建，而建造者负责构建产品的各个部分
