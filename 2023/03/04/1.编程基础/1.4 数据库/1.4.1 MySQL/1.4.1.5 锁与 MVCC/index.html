    1.4.1.5【编程基础 - 数据库 - MySQL】锁与 MVCC

    一、 锁

    1.1 按模式分类

    乐观锁与悲观锁是对待线程竞争的态度，在 Java 语言和 MySQL 数据库中都有此概念的实际应用

    1.1.1 乐观锁

    1.1.1.1 概念

    乐观锁认为当前线程在使用资源的时候，不会（或者说极少有可能）有别的线程来修改资源，所以不会对资源加锁，只是在修改资源的时候，判断是否有其他线程修改了此资源

    1.1.1.2 应用场景

    适用于读多写少的场景，因为在大量写的场景，冲突的可能性极大，业务层可能回不断地重试，十分影响系统性能

    1.1.1.3 实现方式

    一般使用版本号机制实现，在表中增加一个数字类型 version 字段来实现，当更新数据的时候进行 +1

    1.1.2 悲观锁

    1.1.2.1 概念

    悲观锁认为当前线程在使用资源的时候，一定会有其他线程来修改资源，因此在获取资源时会先加锁，确保资源不会被其他线程修改

    1.1.2.2 应用场景

    适用于并发量不大、写入操作比较频繁、数据一致性比较高的场景

    1.1.2.3 实现方式

    在 MySQL 中使用悲观锁，需要关闭 MySQL 的自动提交。共享锁和排他锁是悲观锁的不同实现，也属于悲观锁的范畴

    1.2.2 表级锁

    1.2.2.1 概念

    对整张表进行加锁操作，MyISAM 与 InnoDB 都支持表级锁定

    MySQL 中表级锁有两种，一种是表锁，一种是元数据锁（Meta Data Lock，MDL）

    1.2.2.2 实现方式
    • 表锁：使用 lock tables ... read / write 锁住表，unlock tables 释放锁
      • 读锁 read：当前线程和其它线程只能读
      • 写锁 write：当前线程和其它线程读写都会阻塞
    • 元数据锁：MDL 是隐式的，在 CRUD 时，自动加 MDL 读锁（共享），当修改表结构时，自动加 MDL 写锁（独占）
    1.2.2.3 示例

    console 使用 write 写锁 锁住 goods 表，console_1 读取数据被阻塞

    元数据锁，当 console_1 隐式获得 MDL 读锁时，console 修改表结构的操作需要获取 MDL 写锁被阻塞，console_2 被 console 阻塞了

    解决方案

    • 避免长事务（长事务一般是应用层的锅）
    • 停止 DDL，或者 kill 长事务（谨慎操作）

    1.2.3 页级锁

    1.2.3.1 概念

    页级锁是 MySQL 中锁定粒度介于行级锁和表级锁中间的一种锁。表级锁速度快，但冲突多，行级冲突少，但速度慢。因此，采取了折衷的页级锁，一次锁定相邻的一组记录。BDB 引擎支持页级锁

    1.2.4 行级锁

    1.2.4.1 概念

    行级锁是粒度最低的锁，发生锁冲突的概率也最低、并发度最高。但是加锁慢、开销大，容易发生死锁现象

    1.2.4.2 实现方式

    在 MySQL 中，行级锁并不是直接锁记录，而是锁索引。索引分为 主键索引 和 非主键索引 两种，如果一条 SQL 语句操作了主键索引，MySQL 就会锁定这条主键索引。如果一条语句操作了非主键索引，MySQL会先锁定该非主键索引，再锁定相关的主键索引

    在 UPDATE、DELETE 操作时，MySQL 不仅锁定 WHERE 条件扫描过的所有索引记录，而且会锁定相邻的键值，即所谓的 next-key locking（最下面临键锁有示例）

    1.2.4.3 示例

    console 执行 update 锁住了 id=1 和 id=2 的数据，console_1 更新 id=2 的数据失败，console_2 更新 id=3 的数据成功

    1.3 按属性分类

    1.3.1 共享锁

    1.3.1.1 概念

    共享锁，又称之为读锁，简称 S 锁，当事务 A 对数据加上读锁后，其他事务只能对该数据加读锁，不能做任何修改操作，也就是不能添加写锁。只有当事务 A 上的读锁被释放后，其他事务才能对其添加写锁

    1.3.1.2 应用场景

    共享锁主要是为了支持并发的读取数据而出现的，读取数据时，不允许其他事务对当前数据进行修改操作，从而避免 不可重读 的问题的出现

    1.3.1.3 实现方式

    select ... lock in share mode

    1.3.1.4 示例

    console 查询 id=1 的数据并加 S 锁，console_1 添加写锁失败，更新数据阻塞，console_2 更新数据成功

    1.3.2 排他锁

    1.3.2.1 概念

    排它锁，又称之为写锁，简称 X 锁，当事务对数据加上写锁后，其他事务既不能对该数据添加读写，也不能对该数据添加写锁，写锁与其他锁都是互斥的。只有当前数据写锁被释放后，其他事务才能对其添加写锁或者是读锁

    1.3.2.2 应用场景

    写锁主要是为了解决在修改数据时，不允许其他事务对当前数据进行修改和读取操作，从而可以有效避免 脏读

    1.3.2.3 实现方式

    select ... for update

    1.3.2.4 示例

    console 对 id=1 添加写锁，console_1 获取写锁失败，console_2 获取写锁成功

    1.4 按状态分类

    1.4.1 意向锁概念

    • 意向共享锁（Intention Shared Lock，IS 锁）：事务有意向对表中的某些记录加共享锁（S 锁），加共享锁前必须先取得该表的 IS 锁
    • 意向排他锁（Intention Exclusive Lock，IX 锁）：事务有意向对表中的某些记录加排他锁（X 锁），加排他锁之前必须先取得该表的 IX 锁

    1.4.2 应用场景

    如果需要用到表锁的话，如何判断表中的记录没有行锁呢，一行一行遍历肯定是不行，性能太差。我们需要用到一个叫做意向锁的东西，它记录着表的状态（意向），可以快速判断是否可以对某个表使用表锁

    意向锁之间是互相兼容的

    IS 锁 IX 锁
    IS 锁 兼容 兼容
    IX 锁 兼容 兼容

    意向锁 与 共享锁和排它锁 互斥（这里指的是表级别的共享锁和排他锁，意向锁不会与行级的共享锁和排他锁互斥）

    IS 锁 IX 锁
    S 锁 兼容 互斥
    X 锁 互斥 互斥

    1.4.1.3 示例

    console 执行且不提交的情况下，goods 表存在两把锁，意向排他锁和排他锁。console_1 执行 lock 想获取表 S 锁时，发现 goods 存在意向排他锁，根据上表 S 锁和 IX 锁互斥，console_1 阻塞

    1.5 按算法分类

    记录锁、间隙锁、临键锁都是排它锁，而记录锁的使用方法跟排它锁介绍一致

    1.5.1 记录锁

    记录锁是封锁记录，记录锁也叫行锁，例如 select * from goods where id = 1 for update;，它会在 id=1 的记录上加上记录锁，以阻止其他事务插入，更新，删除 id=1 这一行

    1.5.2 间隙锁

    使用间隙锁锁住的是一个区间，而不仅仅是这个区间中的每一条数据，例如使用 select * from goods where id between 1 and 999 for update; 就可以锁住 id 为 1~999 的数据

    1.5.3 临键锁

    临键锁，是记录锁与间隙锁的组合，它的封锁范围，既包含索引记录，又包含索引区间，是一个左闭右开区间。临键锁的主要目的，也是为了避免幻读。如果把事务的隔离级别降级为 读已提交，临键锁则也会失效

    id name desc price status

    2 盖亚！~ 盖亚！ 77 1
    3 呀哈哈 林克 79 1

    例如上面的数据，price 字段加了普通索引，执行 update goods set name = '只因你~' where price = 77; 时，将会产生隐藏的临键锁：[74, 79)

    二、MVCC

    MVCC 是多版本并发控制方法，即对一份数据会存储多个版本，通过事务的可见性来保证事务能看到自己应该看到的版本。通常会有一个全局的版本分配器来为每一行数据设置版本号，版本号是唯一的

    MVCC 依赖 隐藏字段、Read View、undo log 实现。InnoDB 通过数据行的 DB_TRX_ID 和 Read View 来判断数据的可见性，如不可见，则根据数据行的 DP_ROLL_PTR 找到 undo log 中的历史版本

    2.1 隐藏字段

    在内部，InnoDB 存储引擎为每行数据添加了三个 隐藏字段

    • DB_TRX_ID（6字节）：表示最后一次插入或更新该行的事务 id。此外，delete 操作在内部被视为更新，只不过会在记录头 Record header 中的 deleted_flag 字段将其标记为已删除
    • DB_ROLL_PTR（7字节）：回滚指针，指向该行的 undo log（如果该行未被更新，则为空）
    • DB_ROW_ID（6字节）：如果没有设置主键且该表没有唯一非空索引时，InnoDB 会使用该 id 来生成聚簇索引

    2.2 undo log

    2.2.1 undo log 的作用

    • 当事务回滚时用于将数据恢复到修改前的样子
    • 另一个作用是 MVCC，当读取记录时，若该记录被其他事务占用或当前版本对该事务不可见，则可以通过 undo log 读取之前的版本数据，以此实现非锁定读

    在 InnoDB 存储引擎中 undo log 分为两种：insert undo log 和 update undo log

    2.2.2 insert undo log

    insert undo log 指的是在 insert 操作中产生的 undo log。insert 操作的记录只对当前事务可见，对其他事务不可见，所以在当前事务提交后，insert undo log 可以直接删除

    例如执行以上新增语句，向 test 查询数据，会产生以下 undo log，在事务结束后直接删除。MEM_INDEX 是内存地址（个人假设的，方便理解）

    MEM_INDEX DB_TRX_ID DB_ROLL_PTR id ver
    0x001 1 1 1

    2.2.3 update undo log

    insert undo log 指的是在 update/delete 操作中产生的 undo log，当事务 rollback 时，找到 undo log 中的记录进行回滚。如当前版本对其他事务不可见，可以通过 undo log 读取之前的版本

    假设上述新增已成功提交，test 表包含 id=1,ver=1 的数据，分别开启 3 个事务，执行以上更新语句 3 次并提交事务，此时 test 表的数据和 undo log 如下

    DB_TRX_ID DB_ROLL_PTR id ver
    4 0x003 1 4

    表中隐藏字段 DB_ROLL_PTR 指向 undo log 链表的头结点

    MEM_INDEX DB_TRX_ID DB_ROLL_PTR id ver
    0x003 3 0x002 1 3
    0x002 2 0x001 1 2
    0x001 1 1 1

    2.3 Read View

    MVCC 只在 读已提交（RC） 和 可重复读（RR） 两个隔离级别下工作，而 RC 和 RR 的区别就在于它们生成 ReadView的策略不同。对于使用 RR 和 RC 隔离级别的事务来说，都必须保证读到已经提交了的事务修改过的记录，也就是说假如另一个事务已经修改了记录但是尚未提交，是不能直接读取最新版本的记录的

    核心问题就是：需要判断一下，版本链中的哪个版本是当前事务可见的

    为此，InnoDB 提出了一个 ReadView 的概念，这个 ReadView 中有个 id 列表 trx_ids 来存储系统中当前活跃着的读写事务，也就是 begin 了还未 commit 或 rollback 的事务

    2.4 示例（RR）

    继续用之前的例子来理解一下 ReadView（在 RR 中 select 只会创建一个 Read View） 和 trx_ids。test 表包含 id=1,ver=4 的数据，上次修改它的事务 id 为 4。假设开启两个新事务来执行以下 SQL 语句

    NO 事务 5 事务 6
    1 begin begin
    2 select * from test where id = 1
    3 update test set ver = ver + 1 where id = 1
    4 select * from test where id = 1
    5 commit
    6 select * from test where id = 1
    7 commit

    当事务 5 执行到步骤 2 时，事务 5 执行了查询语句，查出数据为 id=1,ver=4，事务 6 未执行任何操作，此时表数据和 undo log 如下，与之前没什么变化

    DB_TRX_ID DB_ROLL_PTR id ver
    4 0x003 1 4
    MEM_INDEX DB_TRX_ID DB_ROLL_PTR id ver
    0x003 3 0x002 1 3
    0x002 2 0x001 1 2
    0x001 1 1 1

    当事务 5 执行到步骤 4 时，事务 6 执行了更新语句，未提交事务。由于 RR 是可重复读，事务 5 只会读取在它开启之前就已经提事务交的数据，所以事务 5 会查看自己的 ReadView，然后从 undo log 中找到对应的快照。表数据和 undo log 如下

    DB_TRX_ID DB_ROLL_PTR id ver
    6 0x004 1 5
    MEM_INDEX DB_TRX_ID DB_ROLL_PTR id ver
    0x004 4 0x003 1 4
    0x003 3 0x002 1 3
    0x002 2 0x001 1 2
    0x001 1 1 1

    当事务 5 执行到步骤 6 时，表数据和 undo log，即使事务 6 已经提交了，根据可重复度级别，事务 5 会查看自己的 ReadView，然后从 undo log 中找到对应的快照

    2.5 RC 和 RR 的不同

    在 MVCC 中最大的不同就是它们生成 Read View 的时机

    • 在 RC 隔离级别下的 每次 select 查询前都生成一个 Read View
    • 在 RR 隔离级别下只在事务开始后 第一次 select 数据前生成一个 Read View

    如果上面的示例使用的是 RC，那么步骤 2，4 和 6 都会生成一个 Read View

    • 执行步骤 4 时，因为事务 6 未提交所以不可读，在 undo log 中寻找快照返回
    • 执行步骤 6 时，事务 6 已提交了数据，因为 RC 在每次 select 都会生成一个 Read View，所以会读取到事务 6 已提交的数据 id=1,ver=5

    2.6 总结

    从上面的讲解和示例中可以看处，MVCC（Multi-Version Concurrency Control，多版本并发控制）指的就是在使用 RC 和 RR 两种隔离级别中，在执行普通的 select 语句时，访问记录的版本链（undo log）的过程

    这样子可以使不同事务的 读写、写读 操作并发执行，从而提升系统性能

    RC 和 RR 两种隔离级别最大的不同就是生成 Read View 的时机
