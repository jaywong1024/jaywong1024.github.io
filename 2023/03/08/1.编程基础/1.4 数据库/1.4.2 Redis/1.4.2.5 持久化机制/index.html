    1.4.2.5【编程基础 - 数据库 - Redis】持久化机制

    一、持久化分类

    Redis 区别于 Memcached 的一点就是，Redis 支持持久化，并且支持两种不同的持久化操作

    • RDB（Redis Database）：Redis 持久化快照（Snapshotting）方式产出文件的格式 .rdb
    • AOF（Append-Only File）：追加文件方式进行持久化，文件格式为 .aof

    二、RDB 持久化

    2.1 概述

    Redis 可以通过创建快照来获得存储在内存中的数据在某个时间点上的副本。Redis 创建快照后，可以对快照进行备份，可以将快照复制到其他服务器从而创建具有相同数据的服务器副本，还可以将快照保留在本地以便重启时使用

    2.2 自动化持久化

    自动执行快照持久化是 Redis 默认采用的持久化方式，在 redis.conf 配置文件中默认有此配置：

    只要满足以下三个条件之一，BGSAVE 就会被执行

    • 服务器在 900s 内，至少对数据进行了 1 次修改
    • 服务器在 300s 内，至少对数据进行了 10 次修改
    • 服务器在 60s 内，至少对数据进行了 10000 次修改

    2.3 手动持久化

    Redis 提供了以上两个命令来生产 RDB 文件

    • SAVE：主线程执行，会阻塞主线程
    • BGSAVE：子线程执行，不会阻塞主线程

    三、AOF 持久化

    3.1 概述

    相比 RDB，AOF 持久化的实时性更好，是 Redis 主流的持久化方案。Redis 在默认情况下没有开启 AOF 进行持久化，可以通过修改配置文件开启

    3.2 配置

    添加配置 appendonly yes 开启 AOF 持久化后，每执行一条会更改 Redis 中的数据的命令，Redis 就会将该命令写入到内存缓存 server.aof_buf 中，然后再根据 appendfsync 配置来决定何时将其同步到硬盘中的 AOF 文件

    AOF 文件的保存位置和 RDB 文件的位置相同，都是通过 dir 参数设置的，默认的文件名是 appendonly.aof。在 Redis 的配置文件中存在三种不同的 AOF 持久化方式：

    为了兼顾数据和写入性能，用户可以考虑 appendfsync everysec 选项 ，让 Redis 每秒同步一次 AOF 文件，Redis 性能几乎没受到任何影响。而且这样即使出现系统崩溃，用户最多只会丢失一秒之内产生的数据。当硬盘忙于执行写入操作的时候，Redis 还会优雅的放慢自己的速度以便适应硬盘的最大写入速度

    3.3 实现

    AOF 是在 Redis 执行完命令后再往磁盘记录日志，它有两个优点

    • 避免额外的检查开销，AOF 记录日志不会对命令进行语法检查
    • 在命令执行完成后再记录，不会阻塞当前命令的执行

    风险点如下

    • 如果执行完命令后，没来得及写入日志就宕机，会导致对应的修改丢失
    • 可能会阻塞后续的其它命令执行（AOF 记录日志是在 Redis 主线程中执行的）

    四、应该选择哪种方式

    4.1 RDB 优于 AOF 的地方

    • RDB 文件是经过压缩的二进制文件，保存着某个时间点数据库的快照，文件很小，适合做数据备份，灾难恢复。AOF 存储的是每一次写命令，类似于 MySQL 的 binlog 日志，所以 AOF 文件通常会比 RDB 文件大很多
    • 使用 RDB 文件恢复数据，直接解析数据还原即可，不需要像 AOF 重新执行写命令，速度非常快。也就是说，相较于 AOF，RDF 恢复大数据集时更快

    4.2 AOF 由于 RDB 的地方

    • RDB 只有满足一定条件才执行 BGSAVE 命令备份数据，所以 RDB 的数据安全性不如 AOF，没有办法实时或者秒级别的持久化
    • RDB 对于性能的损耗比 AOF 大，RDB 保存整个快照文件，而 AOF 只是追加命令到 AOF 文件，操作轻量
    • AOF 以一种容易理解和解析的格式保存所有操作日志。我们可以轻松的到处 AOF 文件进行分析，如果服务器意外执行了 FLUSHALL 进行删库跑路操作了，还可以使用 AOF 文件进行恢复

    五、新版本对持久化机制的优化

    Redis 4.0 版本开始支持 RDB 和 AOF 混合持久化的方式（默认关闭，可以通过配置 aof-use-rdb-preamble 开启）
