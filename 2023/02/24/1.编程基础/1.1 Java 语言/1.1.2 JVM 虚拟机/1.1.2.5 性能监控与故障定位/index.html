    一、JVM 自带工具

    1.1 jps（JVM 进程状态工具）

    1.1.1 参数

    jps 命令可以查看正在运行 JVM 的进程 ID，可以查看运行包/类名和运行参数等

    参数 说明
    -q 输出本地虚拟机进程的 ID，省略类名称
    -m 输出启动时传递给主类 main() 方法的参数
    -l 输出全类名，如果执行的是 JAR 包，会输出 JAR 包路径
    -v 输出进程的 JVM 配置参数

    1.1.2 例子

    1.2 jstat（状态信息监听工具）

    可以查看 JVM 运行的状态，监听类加载信息、内存信息、垃圾回收信息等

    1.2.1 参数

    参数 说明
    -class 监视类加载、卸载数量、总空间以及类装载的耗时
    -compiler 即时编译器编译过的方法、耗时等信息
    -gc 监视 Java 堆状态，包括 Eden 区、两个 Survivor 区、
    老年代、永久代（JDK 7 后为元空间）的容量、已用空间、垃圾收集时间合计等信息
    -gccapacity 内容与 gc 基本类似，但输出的主要是关注 Java 堆各个区域使用的最大、最小空间
    -gcutil 内容与 gc 类似，输出已使用空间占总空间的百分比
    -gccause 内容与 gcutil 功能一样，但是会额外输出上一次导致垃圾收集产生的原因
    -gcmetacapacity 元空间统计
    -gcnew 监视新生代垃圾收集信息
    -gcnewcapacity 与 gcnew 类似，输出使用到的最大、最小空间
    -gcold 监视老年代垃圾收集信息
    -gcoldcapacity 与 gcold 类似，输出使用到的最大、最小空间
    -printcompilation 输出已经被即时编译的方法

    1.2.2 例子

    1.3 jinfo（配置信息查询工具）

    1.3.1 参数

    jinfo 可以用来查看 JVM 的参数，也可以动态修改运行中的 JVM 参数

    参数 说明
    无参数 输出全部的参数和系统属性
    -flag name 输出对应名称的参数
    -flag [+|-]name 开启或者关闭对应名称的参数
    -flag name=value 设定对应名称的参数
    -flags 输出全部的参数
    -sysprops 输出系统属性

    1.4 jmap（内存映像工具）

    jmap 命令可以生成堆转储快照文件，还可以查询 finalize 执行队列、Java 堆信息、方法去信息等（这些在 JDK 8 后的版本有些不能使用了，所以就不列出参数了），主要看下面 生成堆转储快照文件 的例子

    1.5 jhat（堆转储快照分析工具）

    与 jmap 搭配使用，用来分析 jmap 生成的堆转储快照文件。jhat 内置了一个微信 web 服务器，生成分析后的结果，可以通过浏览器查看。不建议在服务器中使用 jhat，损耗服务器性能，并且在 JDK 8 以后的版本已经被删了，所以这里演示不了了，可以使用 jvisualvm 查看

    这里演示一下用 jvisualvm 打开上面使用 jamp 保存的快照文件 test.hprof 看看。可以看到 Test 内部类的实例为 181 个（拿捏住了，我掐着秒执行的！QAQ）

    1.6 jstack（堆栈跟踪工具）

    jatack 可以生成 JVM 当时的线程快照，它是 JVM 正在执行的方法堆栈集合（所有线程），用于定位线程长时间卡顿的原因（例如死锁、死循环、长时间挂起等），可以排查没响应的线程是什么情况

    故障定位流程依旧使用演示项目，将启动参数中的 -Xms100m -Xmx100m 去除 nohup java -jar spring-framework-1.0.0.jar > /root/spring.date +%Y%m%d.log 2>&1 &

    3.1 CPU

    CPU 异常一般是业务 逻辑问题（例如死循环）或者是 频繁的 gc

    3.1.1 逻辑问题

    可以通过 top 命令找到 CPU 占用较高的进程，再通过 Arthas 找到占用较高的线程，并查看栈帧

    找到占用较高的进程 ID 为 8306

    找到占用较高的线程

    查看线程栈帧，发现是 top.hanjie.web.TestController.kun 的锅

    3.1.2 频繁 gc

    可以通过 Arthas 的 dashboard 命令，查看 gc 信息，并且同步查看 gc 线程的情况，在 [2.2 查看仪表盘](# 2.2 查看仪表盘) 已经演示过了

    3.2 磁盘

    3.2.1 磁盘空间

    通过 df 命令查看磁盘空间占用率

    3.2.2 磁盘 IO 占用

    Centos7 可以安装 iotop 工具（下面有安装截图，Pod 容器可以配置 Dockerfile 安装 iotop），查看 IO 占用，找到 PID 后使用 Arthas 观察磁盘占用较高线程

    重启项目，追加以下接口，用于测试磁盘 IO 占用情况

    调用方法 curl --location --request GET 'http://localhost:8080/test/write?num=2147483647' 后，使用 iotop 命令查看，发现 JVM 线程 ID 为 9234 磁盘 IO 占用最高

    3.3 内存

    内存问题原因比较多，包括 OOM、GC、堆外内存等

    3.3.1 堆内内存

    OOM 相关可以查看上面的 [1.7 模拟 OOM](# 1.7 模拟 OOM)，栈内存溢出 Stackoverflow 一般就是代码问题，例如无限递归，这个通过日志中的异常信息就可以定位到问题代码了（当场逮住，老实改 bug 吧），如果确实需要堆栈较大，可以通过 -Xss2m 来设置（这里设置为 2m，已经是原先两倍了！）

    3.3.2 堆外内存

    堆外内存也是会 OOM 的，例如 Caused by: java.lang.OutOfMemoryError: Meta space，元空间内存溢出，需要配置 XX:MaxMetaspaceSize 设置最大值（一般不会出现）

    其他一般就是和 nio 相关的了（较少遇到，后续遇到再补充）
