    1.1.2.2【编程基础 - Java 语言 - JVM 虚拟机】GC 垃圾回收

    Java 和 C/C++ 不同，它不需要我们手动去申请内存和释放内存，JVM 提供了一套全自动的内存管理机制，当一个 Java 对象被判定为死亡时，JVM 会自动将其进行回收并释放内存。那么，如何去判断对象是否存活，以及如何去进行垃圾回收工作，就是本文关注的问题

    一、对象存活判定算法

    1.1 引用计数算法

    1.1.1 概述

    在 Java 代码中，要操作一个对象，最简单的方法就是使用 new 关键子实例化一个对象，并使用一个变量引用实例对象。如果我们需要在多处代码中操作同一个对象，一般我们会使用多个变量引用此对象。如果我们不需要使用到一个对象，可以将变量赋值为 null。引用计数算法，就是根据对象被引用的数量来判断对象是否存活的，算法如下：

    • 每个对象都包含一个 引用计数器，用于存放引用计数（其实就是存放被引用的次数）
    • 每当有一个地方引用此对象时，引用计数 +1
    • 当引用失效（方法执行完成局部变量不存在了，或者引用变量被设置为 null 了），引用计数 -1
    • 当引用计数为 0 时，表示此对象不可能再被引用，因为这时以及没有任何方法可以得到此对象的引用变量了

    1.1.2 无法解决循环引用

    但是，Java 中没有选用引用计数算法来管理内存，其中最主要的原因是它很难解决对象之间的互相循环引用的问题。例如下面的例子，如果使用引用计数算法将无法回收并释放 3 个 Human 对象占用的内存空间

    1.2 可达性分析算法

    1.2.1 概述

    可达性分析算法 的工作方式：首先构造一个根结点集，用于管理一些称为 GC Roots 的根结点，从这些结点开始，根据引用关系向下搜索，搜索过程的路径称之为 引用链，如果某个对象到 GC Roots 之间没有任何的引用链相连，或者用图论的话来说就是从 GC Roots 到这个对象不可达时，则证明此对象是无法再被使用的

    在 Java 中，成为 GC Root 根结点有以下条件（满足其一即可）

    • 虚拟机栈的栈帧 中的局部变量（例如方法中的局部变量），同样的本地方法栈中 JNI 使用到的对象的引用
    • 方法区 中类的静态属性引用，例如在 Java 类中被 static 修饰的变量
    • 方法区 中常量池中的对象的引用，例如字符串变量
    • JVM 虚拟机内部的引用
    • 被锁（synchronized）持有的对象的引用

    1.3 最终判定（finalize 方法）

    1.3.1 概述

    即使在 可达性分析算法 中判定为死亡的对象也不是 非死不可 的，这些对象暂时处于 缓刑 阶段，要真正宣告一个对象死亡，至少要经历过两次标记过程：

    • 当对象在进行可达性分析算法后发现没有与任何 GC Roots 相连的引用链，那它将会被第一次标记
    • 随后对于被第一次标记的对象再进行一次筛选，判断此对象是否有必要执行 finalize() 方法
      • 无：对象没有覆写 finalize() 方法，或者 finalize() 方法已经被虚拟机调用过了（任何一个对象的 finalize() 方法只会被虚拟机调用一次），那它将会被标记第二次
      • 有：对象被判定为有必要执行 finalize() 方法，那么该对象会被放置在一个名为 F-Queue 的队列中，并在稍后由虚拟机自动建立的、低调度优先级的 Finalizer 线程去执行它的 finalize() 方法。如果执行完 finalize() 方法后依旧没有与任何 GC Roots 相连的引用链，那它将会被标记第二次，否则不会背标记（逃脱 GC）

    对于被标记了两次的对象，基本上它真的就要被回收了

    1.3.2 什么时候应该使用它

    一般用于释放非 Java 资源（比如文件资源、数据库连接等），或是释放非 Java 方法（Native 方法）时分配的内存

    1.3.3 为什么应该避免使用它

    从一个对象被判定为不可达开始，到 finalize() 方法被执行，所花费的时长是任意的，所以 finalize() 方法调用的时机是不确定性的，我们并不能保证 finalize() 方法能够及时的回收资源，可能在资源耗尽之前，gc 却仍未触发，因此通常的做法是提供显示的回收资源的方法供开发者手动调用，例如 java.io.InputStream#close

    因为它的运行代价高昂，不确定性大，无法保证各个对象的调用顺序，如今已被官方明确声明为不推荐使用的语法。finalize() 方法能够完成的工作，使用 try-finally 或者其它方法都可以做得更好、更及时

    二、垃圾回收算法

    在学习了 [一、对象存活判定算法](# 一、对象存活判定算法) 后，已经可以知道堆中的哪些对象满足被回收的条件了，下面就来学习 JVM 是如何进行垃圾回收的

    2.1 分代收集机制

    2.1.1 概述

    在 Java 中不同的对象的生命周期是不一样的（例如一个被 static final 修饰的变量引用的对象，和一个方法中临时 new 出来用于组装字符串的 StringBuilder 对象，它们的生命周期是不一样的），对于不同生命周期的对象都依次判断是否需要垃圾回收，效率是很低下的（因为有些对象在经历多次垃圾回收后依旧是存活的，应该降低扫描此对象的频率）。因此，当前大多数 JVM 都遵循 分代收集 的理论来进行设计，将不同生命周期的对象分别放在不同的区域，针对不同区域的对象使用不同的垃圾回收策略来提高垃圾回收的效率

    JVM 将堆内存划分为 新生代、老年代 和 永久代（其中永久带是 HotSpot 虚拟机特有的概念，在 JDK 8 之前方法区实际上就是采用永久代作为实现，在 JDK 8 之后，方法区由元空间实现，并且使用的是本地内存，容量大小取决于物理机的实际大小）

    不同分代的内存回收机制也存在一些区别，在 HotSpot 虚拟机中，新生代被划分为了三块，分别是一块较大的 Eden 空间和两块小的 Survivor 空间，默认的大小比例为 8:1:1，老年代的 GC 频率相对较低，永久带一般存放类信息（方法区的实现）等，如下图所示：

    2.1.2 GC 分类

    垃圾收集根据不同触发时机和作出以下划分：

    • Minor GC
      • 次要垃圾回收，主要进行新生代的垃圾回收，也称为 Young GC
      • 触发条件：当 Eden 空间容量已满时触发
    • Major GC
      • 主要垃圾回收，主要进行老年代的垃圾回收，也称为 Old GC
      • 触发条件：除了 CMS 和 G1 之外的大多数收集器都没有实现 Major GC，都是直接触发了 Full GC
    • Full GC
      • 完全垃圾回收，对整个堆中的对象和方法去进行垃圾回收
      • 触发条件：
        • 调用 System.gc() 申请执行
        • 老年代空间不足
          • 晋升到老年代的平均内存大小大于老年代可用内存
          • Minor GC 后即将晋升老年代的内存大小大于老年代可用内存
        • 永久代空间不足（JDK 8 之前）

    2.1.4 元空间

    JDK 8 之前，Hotspot 虚拟机中的方法区实际上是存放在 Java 堆内存中的永久代中的。在 JDK 8 之后，Hotspot 虚拟机不再将方法区的数据存放在 Java 堆内存中，而是直接存放在元空间中。元空间使用的是本地内存，不会出现像永久代中的 OOM 错误。这项改造是很有必要的，因为永久代的调优是十分困难的，我们很难确定 Java 类的数量，常量的数量，很难确定一个合适的大小用于容纳方法区，很容易出现 OOM

    2.2 空间分配担保

    2.2.1 概述

    假设 Minor GC 后 Eden 区中存活了大量的对象，而 Survivor 区中存放不下这些存活的对象

    此时就需要老年代进行空间分配担保，将 Survivor 区中无法容纳的对象放到老年代。老年代要进行空间分配担保的前提是有足够的空间来容纳这些来自新生代的对象，因为一共有多少对象需要担保是未知的，所以需要取 之前每次垃圾回收后晋升到老年代的内存大小的平均值 作为参考。使用这个平均值来与老年代可用空间进行比较，来决定是否进行 Full GC 让老年代腾出更多空间

    2.2.2 流程

    • 在发生 Minor GC 之前，虚拟机检测 老年代可用空间 是否大于 新生代所有对象的总内存大小
      • 如果大于，则此次 Minor GC 为安全的
        • Survivor 区可以容纳 Eden 区存活的对象
        • 老年代有足够的内存进行空间分配担保
      • 否则，检查 HandlePromotionFailure = true（是否允许空间担保失败）
        • 允许，则判断 之前每次垃圾回收后晋升到老年代的内存大小的平均值 是否小于 老年代可用空间
          • 小于，则尝试一次 Minor GC（仍然是有风险的）
            • Survivor 区可以容纳 Eden 区存活的对象
            • 老年代有足够的内存进行空间分配担保
            • 老年代装不下，进行 Full GC（依旧内存不足则直接 OOM）
          • 大于，进行 Full GC（依旧内存不足则直接 OOM）
        • 不允许，进行 Full GC（依旧内存不足则直接 OOM）

    2.3 标记-清除 算法

    在上面的 分代收集机制 和 空间分配担保 中，已经大概说明了 堆内存划分规则、GC 触发规则和 整个垃圾回收的运作流程，下面开始了解三个垃圾回收算法，看看在实际的垃圾回收站，不同内存区域使用的回收算法有什么区别

    首先就是 标记-清除 算法，它是最基础的算法

    • 原理：首先，标记出所有需要回收的对象，在标记完成后，统一回收所有被标记的对象
    • 优点：算法简单
    • 缺点：标记清除以后会产生大量不连续的内存空间碎片，内存空间碎片太多可能会导致程序运行过程需要分配较大的对象时，无法找到足够连续的内存空间而不得不提前触发一次垃圾回收
    • 适用：适用于老年代

    2.4 标记-复制 算法

    标记-清除 算法使用后会产生大量不连续的内存空间碎片，而 标记-复制 算法则不会

    • 原理：首先，将可用的内存划分为两块相同的空间，每次只使用其中一块，当使用中的内存用完了，就将内存中还存活的对象复制到另一块空间中，然后把剩余对象清理掉
    • 优点：没有内存空间碎片
    • 缺点：可用内存只有原来的一半
    • 适用：适用于新生代进行垃圾回收（Serial GC、ParNew GC）
    • 优化：由于新生代的对象大多数都会被回收（回收效率高），所以不需要 1:1 的内存划分，可以将内存空间划分为 8:1:1 的 Eden 区、Survivor From 区和 Survivor To 区，每次只使用 Eden 区和 Survivor From 区。当垃圾回收时，将 Eden 区和 Survivor From 区还存活的对象，复制到 Survivor To 区，最后清理掉 Eden 区和 Survivor From 区中的对象。缺点就是当 Survivor From 区空间不够的时候，需要老年代进行空间担保

    2.5 标记-整理 算法

    标记-复制 算法虽然能够处理内存空间碎片问题，但对于回收效率低的区域，属实有点浪费空间了。直接使用 标记-清除 算法后又会产生内存空间碎片。由于需要一种无需浪费 50% 内存空间，又可以整理内存空间碎片的算法，从而产生了结合两者优点的 标记-整理 算法（代价也是有的）

    • 原理：首先，标记所有需要回收的对象，在标记完成后，不急于进行回收操作，而是将所有待回收对象整齐
    • 优点：不会产生内存空间碎片，无需额外内存空间
    • 缺点：它的效率比前两者都低。甚至，因为需要修改对象在内存中的位置，所以会导致程序必须要暂停才可以执行垃圾回收，在极端情况下，可能会导致整个程序发生停顿（被称为 Stop The World）
    • 适用：适用于老年代（Serial Old GC、Parallel Old GC）
    • 优化：可以将 标记-清除 算法和 标记-整理 算法混合使用，在内存空间碎片不是很多的使用，使用 标记-清除 算法，当内存空间比较凌乱后（内存空间碎片较多），可以进行一次 标记-整理 算法来处理内存空间碎片问题

    三、垃圾收集器

    3.1 发展史

    • 1999：第一款垃圾收集器 Serial 和 Serial Old 随着 JDK1.3.1 发布，ParNew 为 Serial 的多线程版本
    • 2002：Parallel Scavenge 和 Parallel Old 与 Concurrent Mark Sweep（CMS）随着 JDK 1.4.2 一起发布（Parallel Scavenge 和 Parallel Old 在 JDK 6 之后成为 Hotspot 默认 GC）
    • 2012：在 JDK 1.7u4 版本中，G1 可用
    • 2017：JDK 9 中 G1 成为默认的垃圾收集器，以代替 CMS
    • 2018.3：JDK 10 中的 G1 垃圾回收器的并行完整垃圾回收功能，实现并行性来改善最坏情况下的延迟
    • 2018.9：JDK 11 发布，引入 Epsilon GC，又被称为 A No-Op GC（无操作垃圾回收期，一个处理内存分配但不实现任何实际内存回收机制的 GC， 一旦可用堆内存用完，JVM 就会退出）。同时，ZGC（可伸缩的低延迟垃圾的回收器）实验版
    • 2019.3：JDK 12 发布，增强 G1，自动返回未用堆内存给操作系统。同时，引入Shenandoah GC（低停顿时间的回收器）实验版
    • 2019.9：JDK 13 发布，增强 ZGC，自动返回未用堆内存给操作系统
    • 2020.3：JDK 14 发布，删除 CMS GC，并扩展 ZGC 在 Mac OS 和 Win 上的应用

    3.2 七款经典 GC 的分类与组合关系

    • 新生代 GC：Serial、ParNew、Parallel Scavenge
    • 老年代 GC：Serial Old、Parallel Old、CMS
    • 整堆 GC：G1

    • 两个收集器之间有连线，则代表它们可以搭配使用，Serial Old 作为 CMS 出现并发故障的后备预案
      • Serial & Serial Old
      • Serial & CMS
      • ParNew & Serial Old
      • ParNew & CMS
      • Parallel Scavenge & Serial Old
      • Parallel Scavenge & Parallel Old
      • G1
    • 红色虚线：由于维护和兼容性测试的成本，在 JDK 8 时将 Serial & CMS、ParNew & Serial Old 这两个组合生命为废弃（JEP 173），并在 JDK 9 中完全取消了这些组合的支持（JEP 214）
    • 绿色虚线：JDK 14 弃用 Parallel Scavenge 和 Serial Old 组合（JEP 366）
    • 深蓝色虚线：JDK 14 删除 CMS GC（JEP 363）

    3.3 什么是 safepoint

    从线程的角度看，safepoint 可以理解成是代码执行过程中的一些特殊位置，当线程执行到这些位置时，说明虚拟机当前的状态是安全的，如果有需要，可以在这个位置暂停，例如发生 GC 时，需要暂停所有活动线程，但是一些线程在此刻还没有执行到安全点，所有这些线程应该继续执行，直到到达安全点的时候暂停，等待 GC 结束

    3.4 七款经典 GC 的实现

    3.4.1 Serial & Serial Old

    Serial GC 是历史最悠久的垃圾收集器，它是一款单线程的垃圾收集器，也就是串行回收收集器。它适用于新生代中，采用 标记-复制 算法，在 GC 时需要暂停所有用户线程（Stop The World），直到 GC 结束。除了新生代，Serial GC 还提供了 Serial Old GC 用于进行老年代的垃圾回收，Serial Old GC 同样也采用串行和 STW 机制

    在 Hotspot VM 中，使用 -XX:+UseSerialGC 可以指定新生代和老年都使用串行收集器

    • 优点：设计简单，高效（在客户端模式可以接受 STW）
    • 缺点：STW 在目前 Java 已微服务架构为主的高并发环境下完全不可用（假设服务每 1h 暂停 5min，成本太高）

    3.4.2 ParNew

    ParNew GC 是新生代垃圾收集器，Par 是 Parallel 的缩写，New 代表只处理新生代。它是 Serial GC 的多线程版本。除了采用并行回收内存外，和 Serial GC 几乎没区别，ParNew GC 同样采用 标记-复制 算法和 STW 机制

    设置 -XX:+UseParNewGC 可以指定新生代使用 ParNew GC（不影响老年代），-XX:ParallelGCThreads 限制 GC 线程数量（默认开启和 CPU 相同的线程数）

    • 优点：在多处理器环境，可以更充分利用资源并行地进行垃圾回收
    • 缺点：STW 机制

    3.4.3 Parallel Scavenge & Parallel Old

    Parallel Scavenge GC 和 ParNew GC 一样，能够并行回收垃圾，同样采用了 标记-复制 算法和 STW 机制。和 ParNew GC 不一样的是，Parallel Scavenge GC 的目标是达到一个可控的吞吐量，它也被称为吞吐量优先收集器。它会自动衡量服务器的吞吐量，并根据吞吐量来决定每次垃圾回收的时间，这种自适应机制，很好的适应了目前微服务架构为主的服务器环境

    Parallel Old GC 用于替代 Serial Old GC，它支持并行回收垃圾，采用了 标记-整理 算法，同样基于 STW 机制

    • 优点：根据吞吐量的自适应机制
    • 缺点：STW 时间虽然缩短，但 GC 频率有可能会上升

    在 JDK 8 中就是采用 Parallel Scavenge GC 和 Parallel Old GC 组合，可以在上面 [1.3.5 执行结果](# 1.3.5 执行结果) 打印的 GC 日志中看出，PSYoungGen 就是 Parallel Scavenge GC 新生代，ParOldGen 就是 Parallel Old 老年代

    3.4.4 Concurrent Mark Sweep（CMS）

    在 JDK 1.5 时期，Hotspot 推出了 CMS GC，它是一款划时代意义的垃圾收集器，这款收集器时 Hotspot 虚拟机中第一个正真意义上支持并发的收集器，它第一次实现了 GC 线程和用户线程同时工作。不过在 G1 GC 问世之后，CMS GC 就不再推荐使用了

    • 优点：并发收集，停顿时间短
    • 缺点：采用 标记-清除 算法会产生内存垃圾碎片，在无法分配大对象的情况下，不得不提前触发 Full GC

    CMS GC 垃圾回收分为 4 个阶段（仅老年代）

    • 初始标记（STW）：这个阶段的主要任务只是标记处 GC Roots 能够关联到的对象，速度比较快，很短的停顿时间
    • 并发标记：从 GC Roots 根结点集合开始，递归扫描出整个对象图，这个过程耗时较长，但是不需要停顿用户线程，GC 线程可以与用户线程一起并发运行
    • 重新标记（STW）：由于在 并发标记 阶段中，用户线程和 GC 线程同时或者交叉与性能，用户线程的运行可能会导致某一些对象的标记产生变动，所以需要对于这些对象进行重新标记，这个阶段的停顿时间通常会比 初始标记 阶段稍长一些，但也远比 并发标记 阶段的时间短
    • 并发清除：子阶段清理掉标记阶段判定已经死亡的对象，释放内存空间（由于不需要移动存活对象，所以这个阶段 GC 线程可以与用户线程一起并发运行

    3.4.5 Garbage First（G1）

    3.4.5.1 概述

    G1 GC 也是一款划时代的垃圾收集器，在 JDK 7 时走上历史舞台，并且在 JDK 9 时，取代了 JDK 8 中默认的 Parallel Scavenge + Parallel Old 方案。G1 GC 与上述的所有 GC 都不同，它跳出分代进行 GC 的约束，将衡量是否进行垃圾回收的标准不再是分代，而是根据 Region（下面会说明）中的垃圾数量最多、回收效益最大来判断

    3.4.5.2 Region

    • G1 GC 将 Java 堆内存分为多个大小相等的 Region，每个 Region 都可以扮演 Eden、Survivor 或者 Old，G1 GC 根据 Region 的角色来采用不同的策略去进行垃圾回收
    • G1 GC 还增加了一种新的内存区域，叫做 Humongous，主要用于存储大对象，如果对象大小超过 1.5 个 Region，就将对象放入 Humongous 中
    • 每个 Region 的大小为 2 的 N 次幂（1MB ~ 32MB），可通过 -XX:G1HeapRegionSize 配置大小（默认由 G1 GC 根据 Java 堆的大小决定）
    3.4.5.3 垃圾回收过程

    当新生代 Eden 区用尽时开始 Minor GC，它会暂停所有用户线程，并启动多个 GC 线程并行执行垃圾回收。G1 GC 会选定所有新生代（Eden 和 Survivor）的 Region，将这些 Region 中存活的对象移动到 Survivor 或者 Old（可能两者都有）的 Region

    当老年代对象占堆大小的一定比例时（默认 45%）时，开始全局并发标记

    • 初始标记（STW）：这个阶段的主要任务只是标记处 GC Roots 能够关联到的对象，速度比较快，很短的停顿时间。在此阶段会触发一次 Minor GC，在 Minor GC 中也有初始标记这一阶段，所以可以不占用额外的停顿时间
    • 并发标记：从 GC Roots 根结点集合开始，递归扫描出整个对象图，这个过程耗时较长，但是不需要停顿用户线程，GC 线程可以与用户线程一起并发运行
    • 再次标记（STW）：修正并发标记的结果，和 CMS GC 类似
    • 独占清理（STW）：计算各个 Region 中存活的对象，并对回收优先级进行排序

    标记完成后马上开始 Mixed GC（混合 GC，是 STW 的），对所有新生代的 Region 和部分 Old Region（根据期望的 GC 停顿时间，以及独占清理的排序结果，确定要清理的 Old GC）以及 Humongous Region

    3.4.5.4 Full GC

    如果上述方式不能正常的工作，G1 GC 会 STW 并使用单线程的内存回收算法进行垃圾回收，性能差并且有较长的停顿时间。触发 Full GC 一般都是因为复制存活对象的时候没有 Region 可用了，所以不建议设置太小的堆内存或者内存较小的服务器使用 G1 GC

    四、引用类型

    4.1 强引用

    在 Java 中，强引用是最普遍使用的引用类型，例如 Object obj = new Object() 中变量 obj 就是强引用了一个 new Object() 对象

    根据上述 [1.2 可达性分析算法](# 1.2 可达性分析算法) 可知，如果某个对象到 GC Roots 之间有引用链相连，那么此对象是无法被判定可回收的，因为它可能还会被使用到。所以，当 JVM 内存空间不足时，JVM 宁愿抛出 OOM 使服务异常停止，也不会靠随意的收回与强引用有相连的对象来解决内存不足的问题

    4.2 软引用

    软引用和强引用不同，当 JVM 认为内存不足时，会试图回收软引用指向的对象，即 JVM 会确保出现 OOM 之前，清理软引用指向的对象。当然，如果内存充足，那么软引用指向的对象是不会被轻易的回收的
