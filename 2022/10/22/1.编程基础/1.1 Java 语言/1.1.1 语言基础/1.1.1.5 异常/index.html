    1.1.1.5【编程基础 - Java 语言 - 语言基础】异常

    参考摘抄：《Effective Java》《Java 核心技术 卷1》

    一、介绍

    1.1 层级关系

    Java 异常结构层次结构简化图，所有异常都是由 Throwable 继承而来。Error 描述了运行时系统内部错误和资源耗尽错误，如果出现这种错误，除了通告给用户，并尽力使程序安全的终止之外，别无他法。Exception 为我们常见的异常，分为 受检异常 checked Exception 和 运行时异常 run-time exception

    1.2 受检异常

    1.2.1 什么是受检异常？

    • 受检异常是强迫处理的，如果不进行处理，将无法通过编译
    • 比如构建 FileInputStream 时，可能会抛出受检异常 FileNotFoundException，必须对其捕获处理或者继续向上层抛出，否则将无法通过编译

    1.2.2 如何判断是否为受检异常？

    • 除了 Error 和 RuntimeException 与其子类，其他都是受检异常

    1.2.3 使用原则

    • 如果期望调用者能够适当地恢复，对于这种情况就应该使用受检异常

    二、使用

    2.1 抛出异常

    使用 thow new 语法即可抛出异常

    2.2 捕获异常

    使用 try catch 代码块即可捕获异常，如需捕获多个异常可以添加 catch 块 or 使用 | 运算符处理

    2.3 再次抛出异常与异常链

    在 catch 子句种可以抛出一个异常，这样做的目的是改变异常的类型

    2.4 finally 子句

    finally 子句一般用于回收资源、释放锁

    2.5 带资源的 try 语句

    对于实现了 java.lang.AutoCloseable 接口的类，可以使用 try-with-resource 自动释放资源，在 try 块退出的时候，会自动调用 java.lang.AutoCloseable#close 方法

    2.6 使用异常的技巧

    2.6.1 异常处理不能代替简单的测试

    对一个空栈进行一百万次的退栈操作，ezTest 方法中使用 if (!stack.empty()) 判断是否可以退栈，exHandel 利用异常捕获机制来告知无法退栈

    打印结果：很显然，ezTest 比 exHandel 快很多

    2.6.2 在检测错误时，“苛刻”要比放任更好

    当检测到错误的时候，有些程序员担心抛出异常。在用无效的参数调用一个方法时，返回一个虚拟的数值，还是抛出一个异常，哪种处理方式更好？例如，当栈空时，Stack.pop 是返回一个 null，还是抛出一一个异常？我们认为：在出错的地方抛出一个 EmptyStackException 异常要比在后面抛出一个 NullPointerException 异常更好

    2.6.3 不要羞于传递异常

    很多程序员都感觉应该捕获抛出的全部异常。如果调用了一个抛出异常的方法，例如，FileInputStream 构造器或 readLine 方法，这些方法就会本能地捕获这些可能产生的异常。其实，传递异常要比捕获这些异常更好
