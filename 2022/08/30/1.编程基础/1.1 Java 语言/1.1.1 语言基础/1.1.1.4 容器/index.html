    四、线程安全

    在 java.util.concurrent 包下都是线程安全的容器，挑几个常用的学习一下

    4.1 ConcurrentHashMap

    4.1.1 简介

    ConcurrentHashMap 是 HashMap 的升级版，HashMap 是线程不安全的，而 ConcurrentHashMap 是线程安全，建议在多线程环境下使用 ConcurrentHashMap

    JDK 1.7 中 ConcurrentHashMap 的数据结构为分段数组链表，如上图所示， ConcurrentHashMap 由很多个 Segment 组成，每一个 Segment 类似于 HashMap 的结构，在并发时对每个 Segment 单独加锁，每个 Segment 内部可以进行扩容。但是 Segment 的个数一旦初始化就不能改变，所以可以认为 ConcurrentHashMap JDK 1.7 默认支持最多 16 个线程并发

    JDK 1.8 的 ConcurrentHashMap 相对 JDK 1.7 来说变化比较大，不再使用 Segment 数组 + 数组链表 作为底层数据结构，而是使用与 HashMap JDK 1.8 一样的数据结构，数组 + 链表/红黑树，对每个 Node 加锁以实现同步

    4.1.2 初始化

    ConcurrentHashMap 通过自旋和 CAS 完成初始化操作，需要注意一下属性 sizeCtl 的状态值，它影响着初始化的进行

    1. 未初始化：sizeCtl = 0（未指定初始化容量）；sizeCtl > 0（指定初始化容量）
    2. 初始化中：sizeCtl = -1
    3. 正常状态：sizeCtl = 0.75*n（表示扩容的阈值）
    4. 扩容中：sizeCtl < 0（表示有线程正在进行扩容）

    4.1.3 写入键值对（JDK 1.8）

    1. 判断是否需要初始化
    2. 通过 hash 值计算下标 i，以 volatile 读的方式获取 table 中的元素，保证每次获取的都是最新的值
    3. 如果 tbale[i] == null，则直接通过 CAS 将新的值封装成 Node 插入即可；如果 CAS 插入失败，说明存在竞争，则进入下一次循环
    4. 如果当前位置的 hashcode == MOVED == -1，则需要进行扩容
    5. 以上都不满足 tbale[i] != null
      1. 先使用 synchronized 将 Node 结点锁住
      2. 判断 Node 结点是链表则遍历链表插入数据
      3. 判断 Node 结点是红黑树则调用 putTreeVal 插入数据

    4.2 ConcurrentSkipListMap

    西瓜游侠的博客-CSDN博客_concurrentskiplistmap

    4.2.1 简介

    ConcurrentSkipListMap 是在 JDK 1.6 中新增的容器，为了对应并发常见下有序 Map 提供更好的支持，它有几个特点：

    • 支持并发场景，是线程同步的
    • 底层是基于跳表 Skip List 实现的
    • key 是有序的
    • key 和 value 都不允许为空

    4.2.2 跳表

    ConcurrentSkipListMap 的底层是基于跳表结构实现的，跳表 Skip List 是一种类似于链表的数据结构，其查询、插入、删除的时间复杂度都是 O(logn)。

    • 跳表由很多层组成
    • 每一层都是一个有序的链表
    • 最底层的链表包含所有元素
    • 对于每一层的任意一个结点，不仅有指向其下一个结点的指针，也有指向下一层的指针
    • 如果一个元素出现在 level == n 的链表中，那它在 n 层以下的层次都会出现

    下图是一种可能的跳表结构：

    如图，[1] 和 [40] 节点有 3 层，[8] 和 [18] 节点有 2 层。每一层都是有序的链表

    如果要查找目标节点 [15]，大致过程如下：

    • 首先查看 [1] 节点的第 1 层，发现 [1] 节点的下一个节点为 [40]，大于 15，那么查找 [1] 节点的下一层
    • 查找 [1] 节点的第 2 层，发现 [1] 节点的下一个节点为 [8]，小于 15，接着查看下一个节点，发现下一个节点是 [18]，大于15，因此查找 [8] 节点的下一层
    • 查找 [8] 节点的第 2 层，发现 [8] 节点的下一个节点是 [10]，小于15，接着查看下一个节点 [13]，小于 15，接着查看下一个节点 [15]，发现其值等于 15，因此找到了目标节点，结束查询

    4.3 CopyOnWrite

    CopyOnWrite 容器，顾名思义就是在写入元素的时候进行复制操作的容器，在向容器添加元素的时候不直接往当前容器添加，而是将当前容器进行拷贝，复制出一个新的容器，然后向新的容器里添加元素，添加元素完成后再将容器原本的引用指向新的容器

    优点

    这样做的好处是可以对 CopyOnWrite 容器进行高效的并发读取，而不需要加锁，因为当前容器不会添加任何元素。所以 CopyOnWrite 容器也是一种读写分离的思想，读和写不同的容器。 CopyOnWrite 通常用于读多写少的场景，例如白名单、缓存数据等

    缺点

    • 内存占用问题：因为 CopyOnWrite 容器写时的复制机制，在写操作时，会占用双倍的内存空间，如果占用的空间比较大很有可能会频繁的造成 Young GC 和 Full GC
    • 数据一致性问题：CopyOnWrite 容器只能保证最终一致性，不能保证实时一致性。当希望写入的数据马上能够读取时，不要使用CopyOnWrite 容器（因为当容器执行 add 或 remove 没完成时，get 获取到的是旧的数据）

    4.3.1 CopyOnWriteArrayList

    CopyOnWriteArrayList 是线程安全且可重复的有序集合，可以看成 ArrayList 线程安全版。在 CopyOnWriteArrayList 的 add 和 remove 方法中可以看到使用了 ReentrantLock 可重入锁保证了线程安全

    4.3.2 CopyOnWriteArraySet

    CopyOnWriteArraySet 是一个线程安全无序不可重复的集合，可以理解为线程安全的 HashSet。不过与 HashSet 不同的是， HashSet 的底层是借助 HashMap 散列表实现的，而 CopyOnWriteArraySet 的底层实现是 CopyOnWriteArrayList，并不是散列表。所以 CopyOnWriteArraySet 也与其他 CopyOnWrite 容器（例如 CopyOnWriteArrayList）一样，适合读多写少的操作，也需要注意内存占用与数据一致性问题

    五、历史容器

    很不幸，许多老的代码是使用 Java1.0/1.1 的容器写成的，甚至有些新的程序也使用了这些类。因此，虽然在写新的程序时，决不应该使用旧的容器，但我们应该了解它们。不过旧的容器功能有限，所以对它们也没太多可以说的

    5.1 Hashtable

    Hashtable 是比较远古使用 Hash 算法的容器了，现在基本已经被淘汰。他的数据结构和 JDK 1.7 的 HashMap 一样，使用的是 数组 + 链表 作为底层数据结构，但和 HashMap 不同的是，Hashtable 是线程安全的，其大多数方法都被 synchronized 修饰，相对效率比较低。所以，在单线程的情况推荐使用 HashMap，多线程的情况推荐使用 ConcurrentHashMap，忘记 Hashtable 吧～

    5.2 Vector

    Vector 是 List 一个典型的实现，底层数据结构和 ArrayList 一样，都是使用 Object[] 数组来保存元素。和 ArrayList 不同的是，Vector 是线程安全的，其大多数方法都被 synchronized 修饰，相对效率比较低。所以，在单线程的情况推荐使用 ArrayList，多线程的情况推荐使用 CopyOnWriteArrayList，忘记 Vector 吧～

    5.3 Stack

    栈是一个 先进后出，后进先出 的数据结构。从 1.0 版本开始，标准类库中就包含了 Stack 类，其中有栈结构典型的 push 和 pop 方法。但是，Statck 作为 Vector 类的扩展，从理论角度看，Vector 类并不台令人满意，它可以使栈使用不属于栈操作的 insert 和 remove 方法，即可以在任何地方都进行插入和删除操作，而不是仅仅在栈顶

    六、常见问题

    6.1 说说 List、Set、Map 三者的区别

    • List：存储的元素是有序的，可重复的
    • Set：存储的元素是无序的，不可能重复的
    • Map：使用键值对进行存储数据，Key 是无序不可能重复，Value 是无序可重复的，每个键最多映射一个值

    6.2 ArrayList 和 LinkedList 的区别

    • 是否线程安全：ArrayList 和 LinkedList 都是不同步的，也就是不保证线程安全的
    • 底层数据结构：
      • ArrayList 底层使用的是 Object[]
      • LinkedList 底层使用的是 Node<E> 双向链表结构（JDK 1.6 之前为循环链表，JDK 1.7 取消了循环）
    • 插入和删除：
      • ArrayList 采用数组存储，插入和删除的时间复杂度受元素位置的影响。例如：执行 add(E e) 方法的时候，ArrayList 会默认将元素插入在列表的末尾，这种情况的时间复杂度就是 O(1)，但是如果要在指定位置 i 插入元素的话 add(int index, E element) 时间复杂度就为 O(n-i)，因为在进行上述操作的时候集合中第 i 个元素和第 (n-i) 个元素都要执行向后/向前位移一位的操作。
      • LinkedList 采用链表存储，所以插入方法和删除元素方法的时间复杂度都不受元素位置的影响，近似 O(1)，如果是要在指定位置 i 插入和删除元素的话 add(int index, E element) 时间复杂度近似为 O(n)，因为需要先移动到指定位置再执行插入/删除操作
    • 是否支持随机访问：
      • ArrayList 支持 get(int index)，它实现了 RandomAccess 接口
      • LinkedList 不支持高效的随机访问
    • 内存占用空间：
      • ArrayList 的空间花费主要在 Object[] 需要预留一些空间
      • LinkedList 的空间花费则体现在每一个元素都保存了额外的信息（用于存储上一个和下一个元素的内存地址）

    • Vector 是线程安全的，其内部方法基本都经过 synchronized 修饰，而 ArrayList 是非线程安全的

    6.5 HashMap 和 Hashtable 的区别是什么？

    • 是否线程安全：HashMap 是非线程安全的，Hashtable 是线程安全的，因为其内部方法基本都经过 synchronized 修饰。如果要保证线程安全，推荐使用 ConcurrentHashMap 吧！
    • 效率：因为线程安全问题，HashMap 要比 Hashtable 高效一些。另外，Hashtable 基本淘汰，不要在代码中使用它
    • 底层数据结构：JDK 1.8 以后 HashMap 为了解决哈希冲突，添加了红黑树作为底层数据结构，Hashtable 与旧的 HashMap 一致
    • 对 null 作为 key 和 value 的支持：
      • HashMap ：支持 null 作为 key 和 value（null 作为 key 只能有一个）
      • Hashtable ：不允许 null 作为 key 和 value，否则会抛出 NullPointerException
    • 初识容量大小与扩容：
      • 创建容器时如果不指定容量初始值，Hashtable 默认为 11，后续的每次扩充，容量变为原来的 2n+1。HashMap 默认的初始值为 16，之后的每次扩充，容量变为原来的两倍
      • 创建容器时指定初始值，Hashtable 就会直接使用给定的大小，而 HashMap 会将其扩充为 2 的幂次方大小（使用 tableSizeFor 进行保证），也就是说 HashMap 总是使用 2 的幂次方作为容器的大小

    6.6 HashMap 和 HashSet 区别是什么？

    HashMap 用于存储键值对，HashSet 用于存储有序不可重复的元素。看 HashSet 的源码可以得知，底层是基于 HashMap 实现的

    6.7 HashSet 如何检查重复？

    在调用 HashSet 的 add 方法时，其实是调用了 HashMap 的 put 方法 public boolean add(E e) { return map.put(e, PRESENT)==null; }，在上面的 HashMap 分析中我们可以知道，put 方法会先计算元素的 hashCode，然后判断根据 hashCode 决定存放在数组中哪一个结点内，再确定了结点后会再使用 equals 方法比较元素是否相等，如果相等则直接覆盖并且返回旧的 value 值，否则在结点中新增数据并返回 null，所以当 HashSet 添加同样的元素时，map.put(e, PRESENT)==null 则会为 false

    简单概括 HashSet 是使用 hashCode 和 equals 来确认重复的，所以当一个对象被 HashSet 使用时，应该覆写他的 hashCode 和 equals 方法，以保证 HashSet 容器的正常运行

    6.8 HashMap 底层实现

    JDK 1.7 的底层实现是 数组 + 链表，JDK 1.8 用的是 数组 + 链表/红黑树

    6.9 HashMap 的长度为什么是 2 的幂次方

    为了提高根据 hashCode 计算数组下标的效率：hashCode 的范围是 -2^31 到 2^31-1，但数组的最大长度为 static final int MAXIMUM_CAPACITY = 1 << 30;，是远小于 hashCode 的，所以在计算数组下标 i 的时候，需要用到取余算法 i = hash % n。重点来了，取余操作中，如果除数是 2 的幂次方则等价于除数减一的与（&）操作，也就是说，当数组长度 n 为 2 的幂次方时，hash % n == (n - 1) & hash，在 Java 中使用二进制的位操作 & 相对 % 能够提高运算效率，所以 HashMap 的长度为 2 的幂次方

    6.10 ConcurrentHashMap 和 Hashtable 的区别

    ConcurrentHashMap 和 Hashtable 主要是底层数据结构和线程安全实现方式上不同

    • 数据结构：

      • Hashtable 使用的是 数组 + 链表 的底层结构
      • ConcurrentHashMap 在 JDK 1.7 使用的是 分段 + 数组 + 链表 的底层结构，在 JDK 1.8 使用的是 数组 + 链表/红黑树 的底层结构

    • 线程安全实现方式：

      • Hashtable 使用 synchronized 修饰了其大部分方法实现了线程安全，效率比较低下。当多个线程同时访问同步方法时，可能会进入阻塞或轮询状态，例如线程 A 在调用 put 方法添加键值对时，线程 B 不能使用 put 和 get 方法，当线程越多竞争则会越激烈，效率则越低
      • ConcurrentHashMap 在 JDK 1.7 时使用 分段锁 实现线程安全，对整个数组分割成了多个 Segment，每个 Segment 都指向一个 数组 + 链表，在调用同步方法时只会对涉及的 Segment 进行加锁，将锁的粒度减小以降低锁竞争的可能性，提高并发访问的效率。到了 JDK 1.8 的时候摒弃了 Segment 的概念，底层数据结构修改为 数组 + 链表/红黑树，使用 synchronized 和 CAS 对数组结点单独加锁来实现线程安全，进一步减少锁的粒度（只有发生 hashCode 冲突的时候才会竞争），效率较 Hashtable 高一些

    6.11 比较 HashSet、LinkedHashSet 和 TreeSet 三者异同

    • HashSet 是一个非同步无序不可重复的集合，数据结构使用 HashMap 实现，允许存在 null 值
    • LinkedHashSet 是 HashSet 的子类，能够按照添加的顺序遍历元素
    • TreeSet 是一个非同步有序不可重复的集合，数据结构使用 TreeMap 实现，插入空值会报错

    6.12 如何选用集合？

    在需要使用到键值对时就使用 Map 下的集合

    • 在需要排序的情况时选择 TreeMap
    • 在不需要排序时就选择 HashMap
    • 在需要保证线程安全且不需要排序时选择 ConcurrentHashMap
    • 在需要保证线程安全且需要排序时选择 ConcurrentSkipListMap

    在需要存放元素且保证元素不可重复时选用 Set 下的集合

    • 在需要排序的情况时选择 TreeSet
    • 在不需要排序时选择选择 HashSet
    • 在需要保证线程安全且不需要排序时选择 CopyOoWriteArraySet（建议用在读多写少场景）
    • 在需要保证线程安全且不需要排序时选择 ConcurrentSkipListSet 底层基于 ConcurrentSkipListMap 实现

    在只需要存放元素且可重复时选用 List 下的集合

    • 在无需线程安全就选择 ArrayList
    • 在需要保证线程安时选择 CopyOnWriteArrayList（建议用在读多写少场景）
